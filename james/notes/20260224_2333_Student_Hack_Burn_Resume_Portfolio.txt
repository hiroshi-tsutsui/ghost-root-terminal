# 【就活・転職】履歴書はゴミ箱へ。GitHubとNotionだけで米国企業に「刺さる」ポートフォリオの作り方【学歴不問】

## 0. はじめに：その「自己分析」で飯は食えん
Jamesです。

いきなりやけど、残酷な現実を言うで。
日本の就活生とか、転職活動中の若手エンジニアが一生懸命やってる「自己分析」と「履歴書の推敲」。
あれ、**米国テック企業の採用基準からしたら、ほぼ「無駄」やねん。**

なんでかわかる？
向こう（US）のPMやエンジニアリングマネージャーは、君の「学生時代の頑張ったこと（ガクチカ）」なんて1ミリも興味ないから。
彼らが知りたいのは**「で、君は何を作れるん？（Show me the code / product）」**。これだけ。

俺も採用に関わることあるけど、TOEICの点数とか大学名なんて見てへん。
**「URL」があるかどうか。**
**そのURLを開いて、3秒で「おっ、使えるな」と思わせるアウトプットがあるか。**

今日は、日本の画一的な「お祈りメール地獄」から抜け出して、**実力だけで年収1000万超えのポジションをハックするための「ポートフォリオ戦略」**を叩き込む。
学歴も英語力も関係ない。「見せ方」というPMスキルがあれば、勝てるんや。

## 1. 「履歴書」というオワコンドキュメント
まず、マインドセットを変えよう。

日本の就活は「減点方式」や。「誤字脱字がないか」「空白期間がないか」。
でも、米国テック企業の採用は**「加点方式」かつ「一発逆転あり」**の世界。

俺が見てきた中で、最強の採用ルートはこれ。
1. **GitHubのリポジトリ**（コードの質＋ドキュメント）
2. **Notionのプロダクトノート**（思考プロセス）
3. **Zenn/Qiita/Noteの技術記事**（発信力）

これらが連携してる「ポートフォリオ」があれば、履歴書の学歴欄が「Fラン」だろうが「高卒」だろうが、面接には呼ばれる。
逆に、東大卒でも「成果物URL」がない奴は、書類選考でAIに弾かれて終わりや。

### 「動くモノ」がないと土俵にすら立てない
「勉強中です」「やる気はあります」
これ、一番言ったらあかんやつな。
テック業界において、**デプロイされてないコードは、書かれてないのと一緒。**
**公開されてない思考は、考えてないのと一緒。**

厳しい？ いや、これが現実や。
逆に言えば、**「ショボくてもいいからデプロイして、URLを送る」**だけで、その他大勢の「勉強中勢」を瞬殺できるってことや。

## 2. 米国PMが「3秒」で判断するポートフォリオの鉄則
じゃあ、具体的にどうすればいいか。
俺が実際に見て「こいつは採用したい」と思ったポートフォリオの共通点を教えるで。

### 鉄則①：READMEがすべて（The First Impression）
GitHubのリンク踏んで、`README.md` がデフォルトのまま、あるいは「一行だけ」の奴。
即ブラウザバック対象や。

PM視点で言うとな、**READMEは「ランディングページ（LP）」**やねん。
ここで「このソフトは何の課題を解決するのか」「どうやって動かすのか」を**英語で**、かつ**図解付きで**説明できてるかどうか。
これができないエンジニアは、「コミュニケーションコストが高い」と判断される。

**【勝てるREADMEの構成】**
1. **Header Image**: キャッチーなロゴかスクショ。
2. **One-Liner**: 1行で「誰の・どんな課題を解決するか」を書く。
3. **Demo GIF**: 実際に動いてる様子をGIFアニメで貼る。（これがないと誰もcloneしてくれん）
4. **Tech Stack**: なぜその技術を選んだかの「選定理由」。
5. **How to Run**: コピペ一発で動くコマンド。

ここで差がつくのが**「Demo GIF」**のクオリティや。
俺はここには金かけろって言ってる。**CleanShot X** 使って、マウスの動きとかクリックのエフェクトまで綺麗に見せるんや。
「神は細部に宿る」。UI/UXへの感度は、こういうスクショ1枚に出るで。

### 鉄則②：Notionで「思考のログ」を残す（The Process）
コードだけじゃ伝わらん「苦労」や「意思決定」を、Notionにまとめてリンクを貼る。
これが実は一番効く。

採用担当が見てるのは「正解を知ってるか」やない。**「未知のバグに当たった時、どう足掻いたか」**や。

- **Why**: なぜこの機能を作ったか（ユーザー視点）
- **Issue**: 開発中にぶつかった壁（技術的負債、パフォーマンス問題）
- **Solution**: どう解決したか（代替案AとBを比較してAを選んだ理由）

この「比較検討（Trade-off Analysis）」のプロセスが書かれてたら、俺なら即採用する。
これができるエンジニアは、現場に入っても「勝手に動いてくれる」からや。PMの負担を減らしてくれる人材、それが一番欲しい人材やねん。

## 3. 「英語が苦手」を武器にする裏技
「でもJamesさん、英語でドキュメント書くとか無理です」
って思ったやろ？

甘い。甘すぎるで。
**英語が下手やからこそ、ドキュメントを書くんや。**

面接で流暢に喋る必要なんてない。
事前にDeepLとChatGPT使って、完璧なREADMEとNotionドキュメントを用意しておく。
面接では**「詳しくはこのNotionに書いてあります。図を見てください」**と言って画面共有する。
これで勝てる。

俺自身、渡米した当初はリスニングなんて壊滅的やった。
でも、図解（Diagram）とドキュメント（Text）で「認識のズレ」を徹底的に潰したから、ネイティブより信頼されたんや。
**「喋りの流暢さ」より「情報の正確さ」。** エンジニアやPMに求められる英語力なんて、中1レベルの文法と、正確な専門用語だけでええねん。

## 4. 今すぐやるべきアクション：『ポートフォリオ駆動開発』
これから何か作るなら、まずコードを書くな。
**「README」から書け。**

これを俺は**「ポートフォリオ駆動開発（PDD）」**と呼んでる。

1. **GitHubにリポジトリを作る。**
2. **README.mdに「理想の完成図（モックのスクショ）」と「解決したい課題」を書く。**
3. **それをSNS（XやLinkedIn）で公開して「これ作るわ！」と宣言する。**
4. **最後にコードを書く。**

この順番でやると、途中で挫折しにくいし、最初から「見せること」を意識した設計になる。
日本のエンジニアは「完成してから公開しよう」とするから、いつまで経っても完成せんのや。
未完成でもええ。「WIP（Work In Progress）」として公開するプロセスそのものが、最強のコンテンツになる。

最後に言わせてくれ。

そんなもんは、AI時代には何の意味も持たん。

君が書いたコード、君が残したドキュメント、君が解決したバグ。
その**「ログ（足跡）」**だけが、君の本当の価値を証明する。

今すぐ履歴書を書く手を止めて、エディタを開け。
GitHubの草（Contributions）を生やせ。
Notionに思考を吐き出せ。

それが、日本の閉塞感から抜け出し、世界基準のキャリアを手に入れる唯一の「裏ルート」や。
