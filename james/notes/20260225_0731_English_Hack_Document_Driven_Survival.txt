発音？ 「This is a pen」が「ディス・イズ・ア・ペン」になるレベルや。

## 1. そもそも「流暢な英語」なんて誰も求めてへん

まず、勘違いを正すで。
日本人は「英語＝流暢に喋ること」だと思い込んでる。
発音がどうとか、仮定法過去完了がどうとか。

**現場ではどうでもええねんそんなこと。**

米国企業の現場、特にエンジニアやPMの世界で求められるのはこれだけや。

1. **Yes / No がはっきりしてるか**
2. **誰が、いつまでに、何をやるかが明確か**
3. **数字が合ってるか**

これだけ。マジでこれだけや。
これ伝えるのに、シェイクスピアみたいな表現いらんやろ？

俺が会議で使う単語なんか、
"Okay", "No problem", "I'll do it", "Check this", "Deadline is tomorrow"
この5つで8割回してる。嘘ちゃうで。

## 2. 俺の武器①：「DeepL」じゃなくて「DeepL Write」を使え

「翻訳ツール使えばええやん」って思うやろ？
甘い。普通のDeepL翻訳やと、日本語のニュアンスが変に訳されて「丁寧すぎる無能」みたいな英語になるんや。

俺が使ってるのは **DeepL Write**（旧：DeepL Write beta）や。
これは「翻訳」やなくて「リライト」ツールや。

俺のやり方はこうや。

1. 自分のクソみたいな英語（中1レベル）を打ち込む。
   `I want you make button red because user easy to see.`
2. DeepL Writeにかける。
3. 提案される。
   `Please make the button red to improve visibility for users.`

これを見て「あ、こういう風に言うんか」って学ぶ。
しかも、AIが提案してくる英語は「シンプルで誤解がない」やつが多い。
ネイティブ独特の言い回しとかいらん。**「小学生でもわかる論理的な英語」**が最強や。

## 3. 俺の武器②：Zoomで喋るな。「Notion」で殴れ

これが最大の極意や。
俺は、重要な意思決定を口頭（Zoom）で絶対にやらん。
なぜなら、聞き取れんし、言いたいことが100%伝わらんからや。

その代わり、会議の前には必ず **Notion** でドキュメントを作る。
これを俺は「ドキュメント駆動開発」ならぬ**「ドキュメント駆動英会話」**と呼んでる。

### 俺の会議スタイル
1. **事前準備:** 議題、背景、俺の提案（Option A / Option B）、それぞれのメリット・デメリットを全部Notionに書く。
2. **図解:** 複雑なロジックは、Mermaid記法かExcalidrawで図にする。「これ見ろ」で終わらせるためや。
3. **会議開始:** "Please read this page first. Let's discuss based on this."（まずこれ読んで。これベースに話そう）と言って、URLを貼る。
4. **沈黙の5分:** 全員が黙って読む。俺はコーヒー飲んでる。
5. **議論:** 全員が内容を理解してるから、質問もピンポイントになる。「Option Aのコストは？」とか。これなら聞き取れる。

このやり方の何が最強かって？
**「英語が下手なこと」がバレへんねん。**
むしろ、「Jamesはいつも準備周到で、ドキュメントがわかりやすい」って評価になる。
ネイティブ同士が口頭で議論して空中戦になってる横で、俺だけが「正解」をテキストで持ってる状態。これが強い。

## 4. 俺の武器③：CleanShot X による「1秒の暴力」

エンジニアへの指示出し、テキストで「右上のメニューの3番目の項目を...」とか書いてないか？
時間の無駄やし、絶対伝わらん。

俺は **CleanShot X** を使い倒してる。
ここでもまた出てきたな、この神ツール。

1. バグってる箇所をスクショ撮る。
2. 矢印（Arrow）と枠（Rect）で囲む。
3. テキストで **"FIX THIS"** とだけ書く。
4. Slackに貼る。

以上や。
これに文法も発音もいらん。
**「画像は世界共通言語」**や。

動画（GIF）も撮りまくる。
再現手順を英語で説明するより、操作してる画面を録画して送った方が100倍早い。
ネイティブのエンジニアも「Jamesのチケットは再現性が高くて助かる」って喜ぶ。
英語力ゼロでも、**「情報の解像度」**が高ければ、お前はチームの英雄になれるんや。
