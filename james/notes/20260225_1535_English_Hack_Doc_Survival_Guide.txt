## 1. 「英語ができる」の定義を履き違えるな

まず、多くの日本人が勘違いしてる「英語ができる」の定義。
これ、だいたいこんな感じちゃいます？

*   ネイティブと流暢に世間話ができる
*   映画を字幕なしで理解できる
*   会議で即座に気の利いたジョークを返せる

**要らん要らん、そんなん1ミリも要らん。**

僕らが目指すのは「通訳」でも「コメディアン」でもない。「プロダクトを前に進めるPM」です。
USテック企業の現場で求められてるのは、流暢な発音じゃなくて**「正確な意思決定」と「明確な指示」**だけなんです。

### 会議でペラペラ喋るやつが偉いわけじゃない

米国の現場にいて気づいたことがあります。
会議でめっちゃ喋るネイティブのPMがおるんですが、彼、実はエンジニアからの信頼が薄いんです。

なんでか？
**「言うことがコロコロ変わるから」**です。

口頭でのコミュニケーションって、その場のノリや勢いに流されやすい。
「I think we should do this!（こうすべきや！）」って情熱的に語っても、翌日には「Wait, actually...（待てよ、やっぱ...）」ってなる。
これ、エンジニアからしたら一番迷惑なやつです。

逆に、僕みたいな英語弱者はどう戦うか。
**口で勝てないなら、文字で勝つんです。**

## 2. 「ドキュメント力」こそが最強の武器

僕の戦い方はシンプルです。
**「全ての意思決定と仕様を、小学生でも分かるレベルのドキュメントにする」。**

これだけ。
でも、これができるPMが意外と少ない。
特にスタートアップはスピード重視やから、「とりあえずZoomで話そうぜ」になりがち。
そこで、**圧倒的に質の高いドキュメント**を叩きつけるんです。

### James流・ドキュメント構成の鉄則

僕が普段書いてるPRD（Product Requirement Document）や仕様書の鉄則を教えます。
英語が苦手な人ほど、これを徹底してください。

#### ① 文章は書くな、箇条書きを使え

**Bad:**
"The user should be able to click the button, and then a modal window will appear, which contains input fields for their name and email address."

（ユーザーはボタンをクリックできるべきで、そうするとモーダルウィンドウが表示され、そこには名前とメールアドレスの入力フィールドが含まれている...）

**Good:**
**User Interaction:**
1. User clicks `[Sign Up]` button.
2. System displays a modal window.
3. Modal contains:
   - `Input: Name`
   - `Input: Email`

分かります？
上の長文、英語苦手な僕らが書こうとしたら文法ミスるし、読む方もダルい。
下の箇条書きなら、**中1英語の単語を並べるだけ**で通じます。
しかも、エンジニアにとっても実装手順が明確で読みやすい。Win-Winです。

#### ② スクリーンショットと図解で黙らせる

「百聞は一見に如かず」は万国共通です。
英語の説明書きなんか、誰も読んでません。
**Figmaのスクショに、矢印引っ張って、そこに「This is MUST」って書く。**
これが最強です。

僕が愛用してるのは**CleanShot X**。
これがないと仕事になりません。
画面キャプチャして、その場で矢印とか文字入れられるMacの神アプリです。
（※ここアフィリエイトリンク貼りたいくらいですけど、今日は我慢します）

「When user clicks here...」って書く暇があったら、クリックする場所に赤丸つけて矢印引っ張る。
これで誤解の余地がなくなります。
**英語力の欠如は、視覚情報量でカバーするんです。**

#### ③ "Definition of Done"（完了の定義）を最初に置く

エンジニアと揉める原因の9割は「認識のズレ」です。
「あれ？ここ実装されてないやん」「え？聞いてないよ」
これを防ぐために、ドキュメントの冒頭に必ず**DoD (Definition of Done)**を書きます。

*   [ ] User can register with email.
*   [ ] Error message appears if email is invalid.
*   [ ] Data is saved to Firestore `users` collection.

これも箇条書き。
チェックボックスにするのがミソです。
これがあれば、会議で英語が聞き取れなくても、「DoDは満たしてるか？」の一点張りで会話が成立します。

## 3. 非同期コミュニケーションの覇者になれ

英語が苦手な僕らにとって、リアルタイムのZoom会議は**「敵のフィールド」**です。
相手の土俵で相撲とっちゃダメです。
僕らの土俵は**「非同期（Asynchronous）」**です。

### Slack/Notionでの戦い方

僕は極力、Zoomを断ります。
「Can we hop on a call?（ちょっと通話できる？）」って言われたら、
食い気味に**「I'm focusing on deep work. Please write it down on Slack/Notion.（今集中してるから、SlackかNotionに書いて）」**って返します。

これ、勇気いりますけど、PMとしては正当な理由です。
「記録に残したいから」と言えば角も立ちません。

テキストなら、**DeepL**が使えます。
Grammarlyで文法チェックもできます。
じっくり推敲して、完璧な英語（に見える文章）を送信できる。

それに、テキストベースでの議論はログが残る。
「言った言わない」の水掛け論になった時、
**「ここに書いてあるやろ（As discussed here...）」**とURLを貼るだけで勝てます。
これが僕の必殺技です。

## 4. エンジニア出身PMの強みを活かせ

もしあなたがエンジニア出身なら、さらに有利です。
**「コードが読める」**というのは、USテック企業において英語力以上の武器になります。

英語の議論についていけなくなったら、黙ってGitHubのPR（Pull Request）を見に行けばいいんです。
コードは嘘をつきません。
どんなに流暢な英語で「実装完了しました！」って言われても、コード見てテスト書いてなかったら「No test, no merge.」って言うだけです。

「I checked the code, logic seems wrong in line 45.」
この一言の威力は絶大です。
**技術的な指摘ができるPMは、英語が下手でもリスペクトされます。**

## まとめ：英語は「ツール」であって「目的」じゃない

長くなりましたが、言いたいことは一つです。

**英語の勉強に時間を使いすぎるな。**
その時間があるなら、ドキュメントの書き方を学べ。
Figmaの使い方を覚えろ。
SQL叩けるようになれ。

USテック企業で求められてるのは、流暢な英語を話す人じゃなくて、
**「プロダクトを成功させるために、誰よりも明確な地図（ドキュメント）を描ける人」**です。

僕の英語は、今でも中学生レベルです。
それでも、年収1000万プレイヤーとして、今日もシリコンバレーのエンジニアたちと渡り合ってます。

**「英語ができないから...」**
そんな言い訳で、自分の可能性を閉ざすのはもう終わりにしませんか？
武器は、あなたの手元にあるキーボードです。
綺麗な英語なんて要らん。
伝わるドキュメントで、世界を殴りにいきましょう。

ほな、また。

James
