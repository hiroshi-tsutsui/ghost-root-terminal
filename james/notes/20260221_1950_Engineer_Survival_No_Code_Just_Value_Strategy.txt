# 【残酷な現実】「コードが書けるだけ」のエンジニアが、年収1000万の壁を一生超えられない理由（と、PMという「裏口」の叩き方）

## はじめに：君の「技術力」は、ビジネスの現場では「1円」にもなっていないかもしれない

「俺は技術が好きやから、コードだけ書いて生きていきたいんです」
「PM？あれは調整屋さんでしょ？興味ないわ」

もし君が今、少しでもそう思ってるなら、この note は劇薬になる。飲むか飲まないかは自由やけど、飲んだら二度と「今のままでいい」とは思えんくなるはずや。

正直に言うてええか？
**シリコンバレーや日本のテック業界で「コードが書けるだけ」の人材の価値は、今、急速に暴落してる。**
AIがコードを書く時代に、人間が「機能の実装」だけで年収1000万、2000万を目指すのは、竹槍で戦闘機に挑むようなもんや。

俺は今、米国のAI企業でプロダクトマネージャー（PM）として働いてる。
年収は1000万を超えてるけど、俺の英語力は「中1レベル」やし、天才的なアルゴリズムが組めるわけでもない。
それでも、現地の天才エンジニアたちが俺の指示で動き、プロダクトが生まれ、売上が立つ。

なぜか？
俺が**「技術（How）」ではなく「価値（Why/What）」を定義できる人間**やからや。

今日は、技術力に自信があるエンジニアこそ陥りやすい「年収の天井」の正体と、そこから抜け出して「技術 × PM」という最強のポジションを取るための**「生存戦略（サバイバルガイド）」**を叩き込む。

これを知ってるか知らんかで、君の5年後の市場価値は「桁」が変わるで。

## 第1章：エンジニアの「市場価値」の正体

### 「機能」を作れる奴は掃いて捨てるほどいる

厳しいこと言うけど、これ現実や。
Reactが書ける、GoでAPIが叩ける、AWSの構築ができる。
素晴らしいスキルや。でもな、それはあくまで**「手段」**なんよ。

ビジネスの現場、特に米国企業の現場で問われるのは一つだけ。
**「そのコードで、いくら稼いだん？」**

これに即答できへんエンジニアは、残念ながら「作業員」扱いで終わる。
作業員である以上、単価は市場相場で決まる。「代わりはいくらでもいる」からや。

### 「1000万の壁」の向こう側にあるもの

年収1000万を超えるプレイヤー（エンジニア含む）に共通してるのは、**「技術を使って、ビジネスの課題を解決した」という実績**や。

- 「この機能を実装しました」 → **×（作業報告）**
- 「この機能を実装して、解約率を5%下げました」 → **△（惜しい）**
- 「解約率を下げるために、技術的に最もコストが低く、かつユーザーに刺さるこの機能を提案し、実装までリードしました」 → **◎（これが1000万プレイヤー）**

わかるか？
コードを書く「前」の、**「何を、なぜ作るのか」**に関与できるかどうかが、勝負の分かれ目なんや。

## 第2章：なぜ多くのエンジニアが「PMへの転身」を恐れるのか？

### 「調整役」という誤解

「PMって、要は雑用係でしょ？」「会議ばっかりしてる人」。
日本企業でよく見る「進捗管理おじさん」を見て、そう思ってるなら大間違いや。

米国の、特にAI企業のPMは**「ミニCEO」**や。
プロダクトの責任を全負いする代わりに、仕様も、デザインも、マーケティングも、全部自分で決められる。
エンジニアに対して「これを作れ」と命令するんやなくて、「なぜこれを作る必要があるか」を熱く語り、彼らを**「その気にさせる」**のが仕事や。

### 技術を捨てるわけではない

「PMになったら、せっかく覚えた技術が錆びつく」
これも嘘や。むしろ逆。
**エンジニア出身のPMこそ、最強になれる。**

なぜなら、エンジニアの痛みがわかるからや。
「この仕様は実装がダルい」「ここをこう変えれば工数が半分になる」。
この感覚（肌感）を持ったまま仕様を切れるPMは、開発チームから神のように崇められる。
**「技術がわかるPM」は、ユニコーン並みに希少価値が高いんや。**

## 第3章：明日からできる「PM OS」のインストール手順

いきなり転職せえとは言わん。
今の現場で、エンジニアのまま「PM的な動き」を試してみるんや。これが一番リスクのない「予行演習」や。

### ① 「How（どう作るか）」の前に「Why（なぜ作るか）」を問う

チケットが降ってきたとき、すぐにエディタを開くな。
そのチケットの起票者（PMやディレクター）に聞きに行け。

「これ、なんのためにやるんでしたっけ？ ユーザーのどんな課題を解決したいんですか？」

ウザがられるかもしれん。でも、これを聞くことで「こいつはビジネス視点を持ってる」と認識される。
そして、もしその仕様がイケてないなら、**「技術的な観点から、もっといい方法がありますよ」と提案する**んや。これがPMへの第一歩や。

### ② ドキュメントで「未来」を語る

コードは「現在」を記述するものやけど、ドキュメントは「未来」を定義するものや。
PMの主戦場は Notion や Google Docs や。

自分が担当する機能について、頼まれてもいないのに**「仕様検討書（Design Doc）」**を書いてみろ。
- 背景（なぜやるか）
- 解決策の案（A案、B案、C案）
- メリット・デメリット
- 推奨案とその理由

これを書いてチームに投げる。
「勝手に決めるな」と怒られるか、「すげえ！これそのまま進めよう」となるか。
後者なら、君はもう実質PMや。

### ③ 「英語」という武器を拾う（逃げるな）

日本市場だけで戦うのは、縮小するパイを奪い合う修羅の道や。
米国企業のPMは、年収のベースが違う。

英語ができない？ 俺もできへんかった。
でもな、**「DeepL」と「Notion」があれば戦える。**
流暢に喋る必要なんてない。
**「図解」と「テキスト」で、誰よりも論理的に仕様を伝える力**があれば、拙い英語でもリスペクトされる。
（この辺の具体的な「英語ハック」は、また別の記事で詳しく書くわ）

## 第4章：James流・キャリアの裏ルート戦略

### 「インターン」やなくて「コントリビューター」になれ

もし君が学生や若手なら、いきなり「PM募集」に応募するな。だいたい経験者採用や。
狙い目は**「エンジニアとして入って、中でPMの仕事を奪う」**パターンや。

1. 小さなスタートアップ（カオスな環境がベスト）にエンジニアとして潜り込む。
2. 仕様がフワフワしてるタスクを率先して拾う。
3. 「これ、仕様詰めときますね」と言って、勝手にPMの仕事を巻き取る。
4. 実績を作ってから、「PMに転向させてくれ」と交渉する。あるいは、その実績を提げて別の会社にPMとして転職する。

これが最短ルートや。**「許可を求めるな、謝罪せよ（Ask for forgiveness, not permission）」**の精神やで。
