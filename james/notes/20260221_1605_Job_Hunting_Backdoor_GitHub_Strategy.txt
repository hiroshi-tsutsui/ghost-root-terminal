# リクナビは捨てろ。無名大学生が「GitHub一本」で米国テック企業の内定をハックする裏ルート

まいど、Jamesです。

今日は**「就活」**の話をします。
特に、今まさにリクナビやマイナビで「エントリーシート」を書いて消耗している大学生、あるいは「未経験からエンジニアになりたい」と思っている社会人2年目くらいの若手に向けて書きます。

結論から言います。
**その「就活」、今すぐやめてください。**
時間の無駄です。
リクルートスーツ着て、髪を黒く染めて、志望動機を暗記して、面接官の顔色を伺う。
そんな「昭和の儀式」に参加している時点で、あなたは**「負け戦」**をしています。

なぜか？
**「優秀なエンジニアやPMは、そんなルートを通らないから」**です。

僕は現在、米国のAI企業でPMをしていますが、同僚のエンジニアたちを見てください。
彼らの多くは、大学の名前なんて誰も知りません。
履歴書（Resume）なんて、入社後に形式的に出すだけです。
彼らが持っているのは、**「圧倒的なGitHubの草（Contributions）」**と、**「自作プロダクトのURL」**だけです。

今日は、日本の画一的な就活システムに絶望しているあなたに、**「学歴もコネもない無名学生が、最短で年収1000万クラスの企業に食い込むための裏ルート」**を授けます。
この記事は長いです。4000文字くらいあります。
でも、読み終わった頃には、あなたの「就活」の概念が180度変わっているはずです。

---\n
## 1. 「許可」を求めるな。「謝罪」をしろ。

日本の就活生は、とにかく**「許可」**を求めます。
「インターンさせてください」
「面接してください」
「話を聞かせてください」

これ、採用側からすると**「コスト」**でしかありません。
忙しいエンジニアやPMが、何の実績もない学生のために時間を割くメリットがゼロだからです。
「やる気はあります！」なんて言葉、1円の価値もありません。
やる気があるなら、もうやってるはずなんです。

米国テック流のハックは真逆です。
**「許可なんて求めるな。勝手にやれ。」**
これが鉄則です。
これを**Permissionless Apprenticeship（許可なき徒弟制度）**と呼びます。

具体的にどうするか？
あなたが入りたい企業（例えば、急成長中のSaaSスタートアップ）があるとします。

1.  その企業のプロダクトを使い倒す。
2.  バグを見つける、あるいは「これがあったら便利だな」という小さな機能（改善案）を考える。
3.  **勝手にコードを書いて、GitHubでプルリクエスト（PR）を送る。**
4.  あるいは、改善案のモックアップ（Figmaやコード）を作って、**CTOやVPoEに直接DMする。**

「勝手にそんなことして怒られませんか？」
怒られたら謝ればいいんです。
でも、まともなテック企業なら、怒るどころか**「驚愕」**します。

「え？ 頼んでもいないのに、バグ直してくれたの？」
「この機能、ちょうど来月実装しようと思ってたやつじゃん！」

この瞬間、あなたは「その他大勢の就活生」から、**「既に仕事をしてくれた仲間（候補）」**に昇格します。
面接なんてただの確認作業になります。
「君、いつから来れる？」
これで終わりです。

## 2. GitHubは「祈りの場」ではない。「武器庫」や。

「GitHubのアカウントは持ってます」
そう言う学生のプロフィールを見ると、大抵がスカスカです。
草（Contributions）が生えているのは、プログラミングスクールの課題をやった日だけ。
リポジトリにあるのは、チュートリアルの写経コードだけ。

ハッキリ言います。
**それはゴミです。**

採用担当が見ているのは、以下の3点だけです。

1.  **「継続性」**: 毎日コードを書いているか？（技術が好きか？）
2.  **「自走力」**: 誰かに言われたからではなく、自分の課題解決のためにコードを書いているか？
3.  **「可読性」**: 他人が読むことを意識したコミットメッセージやドキュメント（README.md）が書かれているか？

特に3つ目が重要です。
米国企業では、非同期コミュニケーションが基本です。
「このコードはどういう意図で書いたの？」と聞かないと分からないコードは、負債です。
READMEに、
-   なぜこれを作ったのか（Why）
-   どうやって動かすのか（How）
-   どんな技術的選定をしたのか（Architecture）
が英語（下手でもいい）で書かれているだけで、評価は**爆上がり**します。

なぜなら、**「ドキュメントが書けるエンジニア」は、世界的に枯渇しているから**です。

### 【悪いREADMEの例】
> `Create ToDo App`
> `Usage: npm start`

これ、何がしたいの？ 誰に向けて書いたの？
採用担当はこれを開いて3秒で閉じます。

### 【勝てるREADMEの構成】
1.  **Title & One-liner**: 「何をするアプリか」を一言で。
2.  **Demo (GIF/Screenshot)**: 動いている様子を見せる（CleanShot Xなどで録画）。これがないと誰もコードを読みません。
3.  **Features**: 主要機能のリスト。
4.  **Tech Stack**: 使用技術（Next.js, Tailwind, Supabaseなど）と、**なぜそれを選んだかの理由**。
5.  **Challenges**: 開発中に苦労した点と、どう解決したか。ここが一番大事。

## 3. 「英語ができない」は言い訳にならん。DeepLを使え。

「でも、米国企業なんて英語ができないと…」
出ました。その言い訳。
僕の記事を読んでくれている人なら分かるはずです。
**英語なんて、DeepLとChatGPTがあればなんとかなる。**

特にエンジニアの場合、コード自体が共通言語です。
PRのコメントやIssueの議論も、テキストベースです。
リアルタイムで流暢に喋る必要なんて、最初のうちは全くありません。

むしろ、
「拙い英語だが、コードは完璧で、ドキュメントも分かりやすい」
これこそが最強の生存戦略です。
「This PR fixes the memory leak issue in the worker process. See benchmark results below.」
これくらいの英語、中学1年生でも書けますよね？

ビビってないで、海外のOSS（オープンソース）にIssueを立ててみてください。
「Typo（誤字）修正」からでいいんです。
世界中のエンジニアと繋がる感覚を一度味わえば、「日本の就活」がいかに狭い世界か痛感するはずです。

## 4. 具体的なアクションプラン：明日からやるべき3つのこと

ここまで読んで「なるほどな」で終わらせないでください。
行動しないと何も変わりません。
明日から、いや今からやるべきことを3つ挙げます。

### ① 自分の「推し」OSSを見つける
有名なライブラリじゃなくていいです。自分が普段使っているツールや、興味のある分野の小さなOSSを探してください。
「Good First Issue」というタグが付いているIssueは、初心者歓迎のサインです。
まずはそこから、「ドキュメントの修正」や「小さなバグ修正」をやってみてください。

### ② ポートフォリオサイトを「英語」で作る
日本語で作ってどうするんですか？ 日本の企業しか見てくれませんよ。
英語で作れば、ターゲット市場が100倍になります。
DeepLで翻訳して、貼り付けるだけです。
そして、そのサイト自体のソースコードも公開し、READMEを充実させてください。

### ③ LinkedInのアカウントを作る
日本の学生はなぜかLinkedInを使いませんが、海外では必須です。
「Open to Work」のタグを付けて、英語でプロフィールを埋めてください。
GitHubのリンクを貼り、作ったものを発信してください。
驚くほどリクルーターから連絡が来ます（スパムも多いですが、たまに本物がいます）。

## 5. 結論：リクルートスーツを燃やせ。PCを開け。

あなたが今やるべきことは、リクナビの登録ボタンを押すことではありません。
マイナビで「合同説明会」の予約をすることでもありません。

1.  **自分が「好きだ」「便利だ」と思うサービスやOSSを見つける。**
2.  **そのソースコードを読む（なければ挙動をハックする）。**
3.  **小さな改善（Fix）を勝手に送りつける。**

これだけです。
これを10回やれば、確率論的に1社くらいは「面白いやつがいる」と返事をくれます。
その1社は、リクナビ経由で入れる企業の100倍、あなたをエンジニアとして成長させてくれる環境のはずです。

**「安定」を求めるな。「実力」をつけろ。**
実力さえあれば、会社が潰れても、国が傾いても、PC一台でどこでも生きていけます。
それが、本当の意味での「安定」です。

さあ、今すぐリクルートスーツを脱いで（あるいは燃やして）、エディタを立ち上げてください。
あなたのキャリアは、あなたが書くコードでしか切り拓けません。
