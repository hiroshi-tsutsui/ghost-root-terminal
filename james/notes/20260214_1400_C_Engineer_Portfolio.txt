**「思考停止」が見透かされるから。**

スクールの課題って、
「ここにこう書けば動きます」って答えがあるやん？
レールの上を走ってるだけやん？

**「おい、コンパスないけど星見て進むぞ」って言える奴**やねん。

### 【実践編】採用担当を唸らせる「README」の魔改造法

まず、ポートフォリオのURLだけ貼って満足してる奴、多すぎな。
採用担当は忙しいねん。
いちいちアプリ開いて、会員登録して…なんてやってくれへん。

勝負は**GitHubのREADME.md**で決まる。

ここで使うべき最強の構成はこれや。
コピペして使ってええで。

```markdown
# [Project Name] - Why I built this?

## 1. The Problem (課題)
- 私は〇〇という課題を持っていました。
- 既存のツールでは××が不満でした。

## 2. The Solution (解決策)
- だから、このアプリを作りました。
- こだわったのは「〇〇」という点です。

## 3. Architecture & Tech Stack (技術選定の理由)
- なぜNext.jsを選んだのか？ → SEOよりUXを重視したから（等）
- なぜFirebaseなのか？ → 開発速度を優先し、認証周りの工数を削減するため。
- **AI Usage:** 
  - ChatGPT: 要件定義の壁打ち、エラーハンドリングの提案
  - Copilot: テストコードの自動生成
  - Midjourney: UIアセットの生成

## 4. Challenges & Learnings (苦労した点と学び)
- 〇〇の実装で詰まりましたが、公式ドキュメントとAIの提案を比較して解決しました。
```

特に見てほしいのは **`3. Architecture & Tech Stack`** の中の **`AI Usage`** や。
これを書くことで、
「あ、こいつはAIをただのコピペマシーンじゃなく、ツールとして使いこなしてるな」
ってアピールできるんや。

### 【極秘】GitHub Copilotへの「英語」指示出しテンプレ

日本の現場やと日本語でコメント書くかもしれんけど、
**年収上げたいなら英語で書け。**
コードもコメントも全部英語や。

なぜか？
**世界中のオープンソースが英語やからや。**
英語で書くだけで「お、こいつ視座高いな」って思われるんやで。

俺がよく使う、Copilotへの指示（コメント）はこれや。

```javascript
// Function: Calculate monthly revenue
// Constraint: Handle edge cases where user data is missing
// Style: Use functional programming approach, avoid side effects
const calculateRevenue = (users) => {
  // Copilotがここに提案してくる
}
```

ポイントは **Constraint（制約）** と **Style（スタイル）** を指定すること。
ただ「書いて」じゃなくて「こう書いて」と指示する。
これが「AIを部下にする」ってことや。
