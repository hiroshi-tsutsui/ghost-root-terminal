# 「PMの記憶喪失」を防ぐ。非エンジニアこそ GitHub で "意思決定" をバージョン管理せよ

「あれ、この仕様ってなんでA案じゃなくてB案になったんでしたっけ？」
「えーっと、3ヶ月前のSlackのどっかで話した気がするんやけど...（検索地獄）」

これ、PMやってて一番胃が痛くなる瞬間やないですか？
僕はこの現象を**「PMの記憶喪失」**と呼んでます。で、これで信頼を失うのが一番しょうもない。

どうも、Jamesです。
米国テック企業でPMやってます。時差16時間のチームと働いてるんで、僕が寝てる間に開発が進むし、向こうが寝てる間に僕が仕様を決める。
つまり、**「なぜそう決めたか」の記録が命**なんです。

今日は、僕がコードを一行も書かずに使っている**「最強の記憶補助装置」**の話をします。
そう、**GitHub** です。

「え、エンジニアのツールでしょ？ 難しそう」
と思ったそこのあなた。**黒い画面（ターミナル）は一切使いません。**
VS Code のボタンをポチポチするだけで、あなたの仕事の「解像度」と「防御力」が爆上がりする話をさせてください。

## なぜ Notion や Google Docs じゃダメなのか？

結論から言うと、**「変更の差分（Diff）」と「その理由（Commit Message）」がセットで残らないから**です。

Google Docs の「変更履歴」機能、あれ使いにくいですよね？
「誰がどこを変えたか」はわかるけど、**「なぜ変えたか」**という意図までは、コメントを辿らないと見えてこない。
Notion も最終形は綺麗に残るけど、**「没になったA案」**は消えてしまうことが多い。

PMの仕事で大事なのは、「現在の仕様」と同じくらい、**「なぜA案を捨てたのか」という死体の山（経緯）**なんです。
これを管理できるのが Git という仕組みです。

## James流「意思決定リポジトリ」の作り方

僕のローカルには `decisions` というフォルダがあります。
中身はただの Markdown ファイルの集まりです。

### Step 1: VS Code をインストールする
騙されたと思って入れてください。Microsoftが作った最強のエディタです。
これに「Git Graph」という拡張機能を入れると、歴史が可視化されて気持ちよくなれます。

### Step 2: "仕様書" ではなく "思考ログ" を書く
例えば `PRD_Login_Feature.md` というファイルを作ります。
最初はこんな感じ。

```markdown
# ログイン機能
- メールアドレス認証にする
- パスワードレス（Magic Link）を採用したい
```

これを「Initial commit」として保存（コミット）します。

### Step 3: 変更するときに "理由" を書く
数日後、エンジニアと話して「Magic Linkはデリバリー率が悪いからやめよう」となったとします。
ここでファイルを書き換えます。

```markdown
# ログイン機能
- メールアドレス認証にする
- ~~パスワードレス（Magic Link）を採用したい~~ -> 廃止
- 従来通りのパスワード認証（12文字以上）
```

そして、コミットメッセージにこう書くんです。
**「fix: Magic LinkはSREチームからの懸念により廃止。SendGridの到達率リスクのため」**

これが大事！！
ファイルの中身（What）だけでなく、コミットログ（Why）に**「SREチームの誰が、何と言ったか」**を残す。

### Step 4: 未来の自分が救われる
半年後。「ねえJames、パスワード入力めんどくさいからMagic Linkにしない？」とCEOが言い出します。
普通のPMなら「えーっと、なんかダメだった気が...」と狼狽えますが、僕は涼しい顔で `VS Code` のタイムラインを見ます。

「あ、それ半年前の議論で、SREのMikeが『SendGridの到達率が98%切るリスクがある』って言ったんで見送ってますね。状況変わったか確認しましょうか？」

**はい、論破（というかFact提示）。** 信頼度爆上がりです。

## 失敗談：最初は「全部」入れようとして死んだ

最初は議事録も、毎日のメモも、全部 GitHub で管理しようとしました。
**これ、失敗します。**
コミットするのが面倒くさすぎて、書くこと自体をやめてしまう。

今は**「意思決定（Decision）」**と**「プロンプト（Prompt）」**だけを入れています。
特に AI へのプロンプト。「先週の出力は良かったのに、今週なんか微妙」って時、プロンプトの `v1` と `v2` の差分が見れるのはデカい。

## まとめ：GitHub は「タイムマシン」や

非エンジニアこそ、Git を使うべきです。
それはコードを管理するためじゃなく、**自分の思考のプロセスを管理するため**。

1. **VS Code を入れる**
2. **`Source Control` タブを使う（コマンドは覚えない）**
3. **「変更理由」をコミットメッセージに書く**

これだけで、あなたは「過去の自分」とチームを組めるようになります。
「記憶」に頼るのはやめましょう。「記録」を、しかも「構造化された記録」を残すんです。

ほな、今日はこの辺で。コミットし忘れるなよ！
