# 【実録】「技術力信仰」を捨てて年収1000万を超えた日。エンジニアこそ今すぐ「PM OS」をインストールせよ

まいど、Jamesです。

今日はちょっと**残酷な現実**の話をします。

特に、今「技術力さえ磨けば年収は上がる」「良いコードを書けば評価される」と信じて疑わない若手エンジニア、あるいは「もっと技術を極めないと生き残れない」と焦っているあなた。

**その「技術信仰」、今すぐ捨てなはれ。**

いや、誤解せんといてください。技術が大事やないと言うてるわけやないんです。僕自身、元々はコードを書いてましたし、技術へのリスペクトはあります。

ただ、**「技術力だけで食っていく」という戦略のROI（投資対効果）が、悲しいほど悪化している**のが今の現実なんです。

特に、僕がいるような米国AI企業の最前線では、その傾向が顕著です。AIがコードを書き、デバッグし、最適化まで提案してくる時代。ここで「人間が書くコードの美しさ」にこだわっても、ビジネスとしての価値は限りなくゼロに近づいています。

じゃあ、どう生き残るか？

答えは一つ。**「技術を武器に持ちつつ、PM（プロダクトマネージャー）というOSを脳にインストールすること」**です。

これが、僕が日本のエンジニア市場で見てきた「年収の天井」をぶち破り、米国AI企業でPMとして生き残っている唯一の戦略です。

今日は、なぜ「技術一本足打法」が危険なのか、そしてどうやって「PM OS」をインストールして**市場価値を一気に跳ね上げるか**、その具体的なロードマップを叩き込みます。

## 1. 「技術力」という名の沼

僕も昔はそうでした。「もっと深い知識があれば」「新しいフレームワークを習得すれば」と、休日は技術書を読み漁り、GitHubに草を生やすことに必死でした。

でも、ある日気づいたんです。

**「これ、どこまで行っても『作業者』の枠を出られへんな」**と。

会社という組織において、**「何を作るか（What）」を決める人間と、「どう作るか（How）」を考える人間では、圧倒的に前者の方がレバレッジが効くんです。** 悲しいかな、給料も権限も、Whatを決める側に集まる構造になっています。

もちろん、超一流のスペシャリスト、例えばGoogleのCoreチームでカーネルをいじってるような天才たちは別です。でも、僕を含めた99%のエンジニアは、そこまでの天才じゃない。

「中の上」くらいの技術力で戦っていると、いつか必ず**「技術のコモディティ化」と「若手の突き上げ」**に挟まれます。そして今、そこに**「AI」という黒船**までやってきた。

この状況で「技術力向上」だけにリソースを全振りするのは、**沈みゆく船の甲板をピカピカに磨いているようなもん**です。

## 2. エンジニアが持つ最強の武器「実装感覚」

ここで朗報です。

実は、エンジニア出身者がPMを目指すとき、**文系出身PMには絶対に真似できない「最強の武器」**を持っています。

それは**「実装感覚（肌感）」**です。

「この機能追加、どれくらい大変そうか？」
「この仕様変更、DB設計にどう影響するか？」
「ここでパフォーマンス問題が起きそうだな」

これらが、直感的にわかる。これ、ものすごいアドバンテージなんです。

多くの非エンジニアPMは、ここがわかりません。だからエンジニアに無茶振りをして嫌われるか、逆にエンジニアに言いくるめられてプロジェクトを遅延させるか、どっちかです。

でも、あなたは違う。

エンジニアと対等に技術の話ができつつ、「ビジネスとして何を優先すべきか」を判断できる。**「通訳」であり「司令塔」になれる。**

このポジションこそが、**現代のテック業界で最も希少性が高く、したがって年収も高い「テクニカルPM（PdM）」**の領域なんです。

## 3. 「PM OS」インストール手順：今日からできる3つのアクション

じゃあ、具体的にどうすればいいのか？ いきなり部署異動願いを出せとは言いません。今の業務の中で、**脳内のOSを書き換える**んです。

### ① 「How」ではなく「Why」から話す癖をつける

エンジニアは、つい「どう実装するか（How）」から話しがちです。「Reactのこのライブラリを使って〜」「DBのスキーマはこうして〜」。

これを、**「なぜそれをやるのか（Why）」**から話すように強制矯正してください。

×「このAPIのレスポンスを速くするためにキャッシュを入れます」
○「**ユーザーの離脱率を下げるために**、表示速度を改善したい。その手段としてキャッシュを検討しています」

主語を「技術」から「ビジネス/ユーザー」に変える。これだけで、周り（特にビジネスサイド）からの見る目が変わります。「こいつ、話がわかるやつやな」と。

### ② "仕様の隙間" を埋めるドキュメントを書く

PMの仕事の8割は**「決まっていないことを決めること」**です。

仕様書に書いてないエラーケース、画面遷移の矛盾、文言の揺れ。これらを見つけた時、Slackで「これどうします？」と聞くだけになってませんか？

そこで一歩踏み込んで、**「現状の課題はA、選択肢はBとCがあるが、工数とUXの観点からBを推奨する」という提案ドキュメント**をサッと書いて投げるんです。

これには**Notion**や**Linear**が最強ですが、ツールは何でもいい。大事なのは**「カオスを構造化して提案する」**という行為です。これができるエンジニアは、実質的にもうPMの仕事をしています。

### ③ 「やらないこと」を決める勇気を持つ

エンジニアは「完璧」を目指しがちです。「あれも対応しなきゃ」「このケースも考慮しなきゃ」。

でも、ビジネスはスピードが命。PM OSでは**「ROI（投資対効果）が低いなら、あえてバグを残す（あるいは対応しない）」**という判断も必要になります。

「この機能、完璧にするのに2週間かかります」ではなく、「**8割の完成度でいいなら3日で出せます。残りの2割はユーザーの反応を見てから決めましょう**」と言えるか。

この**「引き算の思考」**こそが、PMとしてのセンスの分かれ目です。

## 4. 目指すは「コードも読める経営参謀」

僕が米国企業で評価されているのは、英語がペラペラだからではありません（実際、英語は中1レベルで止まってますし）。

評価されているのは、**「エンジニアの痛みがわかり、かつビジネスの数字も追える」というハイブリッドな立ち位置**を確立しているからです。

エンジニアには「技術的に無理なことは言わない信頼できる味方」として。
経営陣には「技術的なリスクとコストを翻訳して説明してくれる参謀」として。

この**「橋渡し（Bridge）」**になれる人材は、日本でも米国でも、まだまだ圧倒的に足りていません。だからこそ、ここを攻めれば**年収1000万なんて単なる通過点**になります。
